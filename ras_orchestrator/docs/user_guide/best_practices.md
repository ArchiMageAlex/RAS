# Лучшие практики

В этом документе собраны рекомендации по проектированию, настройке и эксплуатации RAS-like оркестратора для достижения максимальной эффективности, надёжности и безопасности.

## 1. Проектирование событий

### Ключевые принципы

- **Семантическая ясность**: Поля события должны иметь однозначное значение. Используйте стандартные имена: `severity`, `urgency`, `impact`.
- **Нормализация значений**: Приводите числовые значения к диапазону [0, 1] для упрощения scoring.
- **Контекст в metadata**: Включайте в `metadata` всю дополнительную информацию, которая может понадобиться для принятия решений (идентификаторы пользователей, временные метки, географические данные).
- **Избегайте избыточности**: Не дублируйте информацию в разных полях.

### Пример хорошего события

```json
{
  "event_id": "incident-2026-04-20-001",
  "source": "monitoring",
  "severity": 0.9,
  "urgency": 0.8,
  "impact": 0.7,
  "metadata": {
    "component": "api-gateway",
    "environment": "production",
    "region": "us-east-1",
    "alert_rule": "HighLatency",
    "measured_value": 1200,
    "threshold": 500
  }
}
```

### Пример плохого события

```json
{
  "id": "123",  // неясное имя
  "type": "error",
  "priority": "high",  // строковое значение, сложно сравнивать
  "data": {"foo": "bar"}  // слишком общее поле
}
```

## 2. Настройка Salience Scoring

### Выбор алгоритма

- **Линейная комбинация**: Проста и интерпретируема. Используйте, когда важность факторов известна и фиксирована.
- **ML‑модели**: Применяйте, когда зависимость сложная и есть исторические данные для обучения. Начинайте с простых моделей (логистическая регрессия), затем переходите к более сложным (градиентный бустинг, нейросети).
- **Ансамбли**: Комбинируйте несколько алгоритмов для повышения устойчивости.

### Калибровка весов

1. Соберите исторические события с экспертной оценкой важности.
2. Настройте веса с помощью оптимизации (например, grid search) для минимизации ошибки.
3. Регулярно перекалибровывайте веса при изменении паттернов событий.

### Мониторинг качества scoring

- Отслеживайте распределение salience score (`ras_salience_score_distribution`).
- Сравнивайте предсказанные score с фактическими исходами (если есть обратная связь).
- Настройте алерты на аномалии (например, если все события внезапно получают score 0 или 1).

## 3. Проектирование политик

### Структура политик

- **Одна политика — одна ответственность**: Избегайте монолитных политик, которые делают много всего. Разделяйте на мелкие, композируемые политики.
- **Используйте DSL**: Политики, написанные на DSL, легче читать, тестировать и версионировать, чем код.
- **Добавляйте описания**: Каждая политика должна иметь поле `description`, объясняющее её назначение.

### Условия (Conditions)

- Делайте условия простыми и тестируемыми.
- Избегайте сложных логических выражений с множеством `and`/`or`. Если условие становится сложным, разбейте его на несколько политик.
- Используйте функции из DSL для проверки временных интервалов, строковых паттернов и т.д.

### Действия (Actions)

- Действия должны быть идемпотентными (повторное выполнение не должно ломать систему).
- Всегда предусматривайте откат (rollback) для деструктивных действий.
- Логируйте выполнение действий с достаточным контекстом.

### Пример хорошей политики

```yaml
name: notify-on-high-severity
description: "Отправлять уведомление в Slack при событиях с severity > 0.9"
condition: event.severity > 0.9
action: webhook
action_params:
  url: "https://hooks.slack.com/services/..."
  method: POST
  headers:
    Content-Type: "application/json"
  body: |
    {
      "text": "High severity event: {{ event.event_id }}",
      "severity": "{{ event.severity }}"
    }
```

### Тестирование политик

- Создайте набор тестовых событий для каждого сценария.
- Используйте API Policy Engine для валидации политик перед применением.
- Внедрите политики постепенно: сначала в мониторинге, затем в warning‑режиме, и только потом в production.

## 4. Управление режимами (Mode Management)

### Настройка гистерезиса

Гистерезис предотвращает колебания режимов при borderline значениях salience score.

- **Пороги перехода**: Установите разные пороги для перехода вверх и вниз. Например:
  - normal → elevated: score > 0.7
  - elevated → normal: score < 0.5
- **Временные задержки**: Добавьте минимальное время пребывания в режиме перед переходом.

### Cooldown периоды

После critical режима введите cooldown период, в течение которого система не может снова перейти в critical, даже если score высокий. Это даёт время на восстановление.

### Мониторинг режимов

- Отслеживайте метрику `ras_current_mode`.
- Настройте алерты на длительное пребывание в critical режиме.
- Визуализируйте историю переходов на дашборде.

## 5. Обработка прерываний

### Типы прерываний

- **Soft interrupt**: Отложенное прерывание, которое позволяет завершить текущие задачи. Используйте для non‑critical событий.
- **Hard interrupt**: Немедленное прерывание. Применяйте только когда задержка недопустима.
- **Delayed interrupt**: Прерывание с фиксированной задержкой. Полезно для batch‑обработки.

### Чекпоинты

- Сохраняйте состояние перед каждым прерыванием, чтобы можно было восстановиться.
- Храните чекпоинты в Redis с TTL, чтобы избежать накопления.
- Реализуйте механизм очистки старых чекпоинтов.

### Восстановление

- После прерывания система должна иметь возможность продолжить с чекпоинта.
- Протестируйте восстановление при сбоях (например, убийство процесса).

## 6. Производительность и масштабирование

### Оптимизация Kafka

- **Partitioning**: Разделяйте топики на достаточное количество partitions для параллельной обработки.
- **Compression**: Включите сжатие (snappy, gzip) для уменьшения сетевого трафика.
- **Batch size**: Настройте batch size потребителей для баланса между latency и throughput.

### Redis

- **Connection pooling**: Используйте пул соединений для уменьшения overhead.
- **Pipelining**: Группируйте команды в pipeline для снижения latency.
- **Memory management**: Установите maxmemory и политику вытеснения (например, allkeys‑lru).

### База данных (PostgreSQL)

- **Индексы**: Создайте индексы на часто запрашиваемых полях (`event_id`, `created_at`, `status`).
- **Partitioning**: Разделяйте большие таблицы (например, события) по времени.
- **Connection pool**: Настройте пул соединений на уровне приложения (например, через asyncpg).

### Горизонтальное масштабирование

- Каждый компонент должен быть stateless или хранить состояние во внешнем хранилище (Redis, DB).
- Используйте Kubernetes HPA для автоматического масштабирования на основе метрик (CPU, Kafka lag, RPS).
- Размещайте компоненты в разных availability zones для отказоустойчивости.

## 7. Наблюдаемость (Observability)

### Логирование

- Используйте структурированные логи (JSON).
- Включайте correlation ID в каждое событие, чтобы отслеживать его через все компоненты.
- Настройте разные уровни логирования для production (INFO) и разработки (DEBUG).

### Метрики

- Экспортируйте бизнес‑метрики (количество событий, salience score, режимы) и системные метрики (latency, ошибки).
- Определите SLO/SLA и настройте алерты на их нарушение.
- Используйте гистограммы для latency, чтобы видеть распределение, а не только среднее.

### Трассировка

- Инструментируйте все компоненты с OpenTelemetry.
- Убедитесь, что трассировки передаются между асинхронными границами (Kafka).
- Настройте sampling для уменьшения объёма данных (например, 10% трассировок).

## 8. Безопасность

### Аутентификация и авторизация

- Используйте API‑ключи для доступа к API Gateway. Храните ключи в secrets manager (Hashicorp Vault, AWS Secrets Manager).
- Реализуйте ролевую модель (RBAC) для управления политиками.
- Все внутренние коммуникации должны использовать mTLS.

### Защита данных

- Шифруйте данные в покое (диски БД, Redis persistence).
- Шифруйте данные в движении (TLS для Kafka, Redis, PostgreSQL).
- Не храните чувствительные данные в логах (маскируйте пароли, токены).

### Безопасность контейнеров

- Используйте минимальные базовые образы (например, `python:3.10‑slim`).
- Запускайте контейнеры от non‑root пользователя.
- Регулярно сканируйте образы на уязвимости (Trivy, Snyk).

## 9. Резервное копирование и восстановление

### Что备份ить

- **PostgreSQL**: Полный дамп + WAL‑логи.
- **Redis**: RDB snapshot + AOF.
- **Kafka**: Критичные топики через mirror maker.
- **Конфигурации**: Git репозиторий с политиками и манифестами.

### Частота

- PostgreSQL: ежедневно полный дамп, каждые 5 минут WAL.
- Redis: каждый час snapshot.
- Kafka: непрерывная репликация в другой кластер.

### Тестирование восстановления

- Регулярно (раз в месяц) проводите учения по восстановлению из backup.
- Измеряйте время восстановления (RTO) и корректность данных (RPO).

## 10. Управление инцидентами

### Runbooks

Создайте runbooks для типичных сценариев:

- Высокий Kafka lag.
- Redis недоступен.
- Все события получают salience score 0.
- Система застряла в critical режиме.

### Коммуникация

- Определите каналы уведомлений (Slack, PagerDuty, SMS).
- Назначьте ответственных за каждый тип инцидента.
- Проводите постмортемы для каждого серьёзного инцидента и обновляйте документацию.

## 11. Обновления и миграции

### Стратегия обновления

- Используйте blue‑green или canary deployment для минимизации downtime.
- Тестируйте обновления в staging срежде, чем в production.
- Имейте план отката на предыдущую версию.

### Миграции данных

- Все миграции БД должны быть идемпотентными и обратно совместимыми.
- Выполняйте миграции во время низкой нагрузки.
- Создавайте backup перед миграцией.

## 12. Документация и знания

### Поддерживайте актуальность

- Документируйте все изменения в архитектуре, API и политиках.
- Используйте автоматическую генерацию документации (например, из OpenAPI спецификации).
- Проводите регулярные обзоры документации с командой.

### Обучение

- Создайте тренировочные сценарии для новых членов команды.
- Проводите workshops по написанию политик и диагностике проблем.
