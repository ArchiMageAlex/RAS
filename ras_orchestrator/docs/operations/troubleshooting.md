# Руководство по устранению неполадок

В этом документе описаны распространённые проблемы, их диагностика и решения.

## Общий подход

1. **Определить область проблемы**: какой компонент затронут (API Gateway, Salience Engine, Kafka, Redis и т.д.).
2. **Собрать данные**: логи, метрики, трассировки.
3. **Проанализировать**: найти root cause.
4. **Применить решение**: временное или постоянное.
5. **Документировать**: обновить runbook.

## Инструменты диагностики

### Логи

Логи всех компонентов собираются в Loki. Для поиска:

```bash
# Просмотр логов через kubectl
kubectl logs deployment/api-gateway -n ras --tail=100

# Поиск в Loki через LogCLI (если установлен)
logcli query '{app="api-gateway"}' --limit=100 --from=1h

# Просмотр логов через Grafana (Loki datasource)
```

### Метрики

Prometheus предоставляет метрики. Используйте Grafana для визуализации или прямые запросы:

```bash
# Порт-форвард Prometheus
kubectl port-forward svc/monitoring-prometheus 9090:9090 -n monitoring

# Затем запрос через браузер: http://localhost:9090
```

### Трассировки

Распределённые трассировки хранятся в Jaeger. Доступ через порт-форвард:

```bash
kubectl port-forward svc/monitoring-jaeger-query 16686:16686 -n monitoring
```

Откройте http://localhost:16686 для поиска трассировок.

### Проверка состояния инфраструктуры

```bash
# Проверить состояние подов
kubectl get pods -n ras

# Проверить состояние Kafka
kubectl exec -it deployment/kafka -n ras -- kafka-topics --list --bootstrap-server localhost:9092

# Проверить Redis
kubectl exec -it deployment/redis -n ras -- redis-cli ping

# Проверить PostgreSQL
kubectl exec -it deployment/postgres -n ras -- psql -U ras -d ras -c "SELECT 1;"
```

## Распространённые проблемы

### 1. События не поступают в систему

**Симптомы**:
- Метрика `ras_events_total` не растёт.
- В логах API Gateway нет записей о новых событиях.
- Клиенты получают ошибки при отправке событий.

**Возможные причины**:
- API Gateway недоступен.
- Проблемы с сетью или балансировщиком.
- Ошибки аутентификации (неверный API‑ключ).
- Превышен rate limit.

**Диагностика**:
1. Проверить health endpoint: `curl http://api-gateway:8000/health`
2. Проверить логи API Gateway на наличие ошибок.
3. Убедиться, что Kafka consumer (salience‑engine) работает и имеет lag.
4. Проверить конфигурацию API‑ключа в заголовке `X-API-Key`.

**Решение**:
- Перезапустить API Gateway: `kubectl rollout restart deployment/api-gateway -n ras`
- Проверить NetworkPolicy и Ingress.
- Увеличить rate limit, если необходимо.

### 2. Высокий Kafka consumer lag

**Симптомы**:
- Метрика `kafka_consumer_lag` растёт.
- Задержка обработки событий увеличивается.
- В логах consumer'ов сообщения о медленной обработке.

**Возможные причины**:
- Потребитель (например, salience‑engine) работает медленно.
- Недостаточно реплик consumer.
- Проблемы с сетью между Kafka и consumer.
- Большой размер сообщений.

**Диагностика**:
1. Проверить метрики CPU/памяти consumer.
2. Посмотреть логи consumer на наличие ошибок или предупреждений.
3. Проверить количество partitions топика и распределение.
4. Использовать `kafka-consumer-groups` для детального анализа lag.

**Решение**:
- Увеличить количество реплик consumer.
- Оптимизировать логику обработки (увеличить batch size, уменьшить сложность).
- Настроить auto‑scaling на основе lag.
- Увеличить ресурсы (CPU, память) для consumer.

### 3. Redis ошибки соединения или таймауты

**Симптомы**:
- В логах компонентов ошибки типа `RedisTimeoutError`, `ConnectionError`.
- Увеличивается задержка операций с Redis.
- Метрика `redis_operations_failed_total` растёт.

**Возможные причины**:
- Redis перегружен (высокая загрузка CPU/памяти).
- Сетевые проблемы.
- Достигнут лимит соединений.
- Проблемы с persistence (RDB/AOF).

**Диагностика**:
1. Проверить метрики Redis: использование памяти, количество подключений, latency.
2. Проверить логи Redis на наличие предупреждений.
3. Использовать `redis-cli --stat` для мониторинга в реальном времени.

**Решение**:
- Увеличить ресурсы Redis (память, CPU).
- Настроить connection pooling в клиентах.
- Включить мониторинг slow queries и оптимизировать.
- Рассмотреть использование Redis Cluster для распределения нагрузки.

### 4. Salience score всегда 0 или 1

**Симптомы**:
- События получают экстремальные salience score (0 или 1), что приводит к неадекватным решениям.
- В логах salience‑engine нет ошибок.

**Возможные причины**:
- Неправильная конфигурация модели или весов.
- Проблемы с входными данными (отсутствие обязательных полей).
- Баг в алгоритме scoring.

**Диагностика**:
1. Проверить конфигурацию scoring в `salience_engine/scoring.py`.
2. Убедиться, что события содержат поля `severity`, `urgency`, `impact`.
3. Посмотреть логи salience‑engine на наличие предупреждений.
4. Проверить, что ML‑модель (если используется) загружена корректно.

**Решение**:
- Обновить конфигурацию весов.
- Добавить валидацию событий на входе.
- Переобучить/обновить ML‑модель.
- Временно переключиться на базовый scoring.

### 5. Частые переходы между режимами (mode thrashing)

**Симптомы**:
- Метрика `ras_mode_transitions_total` быстро растёт.
- Система постоянно переключается между normal и elevated.
- В логах mode‑manager много записей о переходе.

**Возможные причины**:
- Слишком низкие гистерезисные пороги.
- События с колеблющимся salience score.
- Агрессивные политики.

**Диагностика**:
1. Проверить конфигурацию гистерезиса в `mode_manager/manager.py`.
2. Проанализировать salience score событий за период thrashing.
3. Проверить политики mode transition.

**Решение**:
- Увеличить гистерезисные пороги (например, с 0.1 до 0.2).
- Добавить cooldown период после перехода.
- Настроить smoothing salience score (скользящее среднее).

### 6. Задачи не создаются или зависают

**Симптомы**:
- Метрика `ras_tasks_created_total` не растёт.
- Задачи в статусе pending долгое время.
- Агенты не получают задачи.

**Возможные причины**:
- Task Orchestrator недоступен.
- Проблемы с Redis (очередь задач).
- Ошибки валидации задач.
- Недостаточно агентов.

**Диагностика**:
1. Проверить health task‑orchestrator.
2. Убедиться, что Redis очередь (`ras:tasks:pending`) существует.
3. Проверить логи task‑orchestrator на наличие ошибок.
4. Убедиться, что retriever‑agent подключён и потребляет задачи.

**Решение**:
- Перезапустить task‑orchestrator.
- Очистить очередь задач (если она забита).
- Увеличить количество агентов.
- Проверить конфигурацию политик создания задач.

### 7. Высокая задержка обработки событий

**Симптомы**:
- Метрика `http_request_duration_seconds` показывает высокие значения.
- Клиенты жалуются на медленный ответ.
- Трассировки показывают длительные этапы.

**Возможные причины**:
- Перегрузка какого‑либо компонента.
- Медленные внешние зависимости (Kafka, Redis, ML‑модель).
- Неоптимальные настройки пулов потоков/соединений.

**Диагностика**:
1. Использовать Jaeger для выявления узких мест.
2. Проверить метрики latency каждого компонента.
3. Проанализировать CPU/память.

**Решение**:
- Оптимизировать медленные операции (кеширование, batch processing).
- Увеличить ресурсы компонентов.
- Настроить горизонтальное масштабирование.
- Использовать асинхронную обработку где возможно.

### 8. Ошибки политик (Policy Engine)

**Симптомы**:
- В логах policy‑engine ошибки парсинга или выполнения политик.
- Политики не применяются к событиям.
- Веб‑интерфейс политик не загружается.

**Возможные причины**:
- Невалидный YAML в файле политик.
- Ошибки в DSL (синтаксис, неизвестные функции).
- Проблемы с доступом к БД политик.

**Диагностика**:
1. Проверить логи policy‑engine.
2. Протестировать политики через API: `POST /api/v1/policies/validate`.
3. Убедиться, что файлы политик загружены корректно.

**Решение**:
- Исправить синтаксис YAML.
- Обновить DSL или добавить отсутствующие функции.
- Перезагрузить политики через API.

### 9. Проблемы с Observability стеком

**Симптомы**:
- Grafana дашборды не обновляются.
- Prometheus не собирает метрики.
- Loki не возвращает логи.
- Jaeger не показывает трассировки.

**Возможные причины**:
- Недостаточно ресурсов для monitoring стека.
- Проблемы с persistent volume.
- Конфигурационные ошибки.

**Диагностика**:
1. Проверить поды в namespace monitoring.
2. Проверить логи Prometheus, Grafana, Loki.
3. Убедиться, что service discovery работает.

**Решение**:
- Увеличить ресурсы monitoring стека.
- Проверить и исправить конфигурацию.
- Пересоздать повреждённые PVC.

## Экстренные процедуры

### Откат версии

Если новая версия вызывает проблемы, выполните откат:

```bash
# Для Kubernetes
kubectl rollout undo deployment/api-gateway -n ras

# Для Docker Compose
docker-compose down
git checkout previous-version-tag
docker-compose up -d
```

### Отключение компонента

Временно отключить проблемный компонент, чтобы восстановить работу системы:

1. Масштабировать до 0 реплик: `kubectl scale deployment api-gateway --replicas=0 -n ras`
2. Направить трафик в обход (если возможно).
3. После исправления вернуть реплики.

### Ручное управление режимами

Если mode‑manager работает некорректно, можно принудительно установить режим:

```bash
curl -X POST http://api-gateway:8000/api/v1/mode/force \
  -H "X-API-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"mode": "normal", "reason": "manual override"}'
```

### Очистка очередей

Если Kafka или Redis очереди переполнены, можно очистить их (осторожно!):

```bash
# Очистить топик Kafka
kubectl exec -it deployment/kafka -n ras -- kafka-topics --delete --topic ras.events --bootstrap-server localhost:9092

# Очистить Redis ключи
kubectl exec -it deployment/redis -n ras -- redis-cli FLUSHALL
```

## Профилактика

1. **Регулярные проверки**:
   - Ежедневный обзор дашбордов.
   - Проверка алертов.
   - Анализ трендов метрик.

2. **Тестирование отказоустойчивости**:
   - Периодически вызывать сбои (chaos engineering) в staging среде.
   - Тестировать восстановление после потери компонентов.

3. **Обновление документации**:
   - После каждого инцидента обновлять этот troubleshooting guide.
   - Добавлять новые сценарии и решения.

4. **Тренировки команды**:
   - Проводить учения по реагированию на инциденты.
   - Отрабатывать процедуры восстановления.
