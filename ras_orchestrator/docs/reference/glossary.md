# Глоссарий терминов

В этом документе приведены определения ключевых терминов, используемых в RAS-like оркестраторе.

## A

**API Gateway**  
Компонент, который принимает входящие HTTP запросы (события) от клиентов, выполняет аутентификацию, валидацию и публикует события в Kafka.

**Action (Действие)**  
Операция, выполняемая в результате срабатывания политики. Примеры: отправка webhook, создание задачи, изменение режима.

**Alert (Алерт)**  
Уведомление о нарушении метрики или условия, которое может быть отправлено в RAS как событие.

## B

**Backpressure (Обратное давление)**  
Механизм, предотвращающий перегрузку системы путём замедления обработки, когда downstream компоненты не успевают.

**Batch Processing (Пакетная обработка)**  
Обработка группы событий за один раз для повышения эффективности.

## C

**Checkpoint (Чекпоинт)**  
Снимок состояния системы в определённый момент времени, позволяющий восстановиться после прерывания или сбоя.

**Cooldown (Охлаждение)**  
Период после critical режима, в течение которого система не может снова перейти в critical, даже если salience score высокий.

**Correlation ID (Идентификатор корреляции)**  
Уникальный идентификатор, присваиваемый событию при поступлении и передаваемый через все компоненты для трассировки.

**Critical Mode (Критический режим)**  
Режим работы системы, в котором обрабатываются только высокоприоритетные события, а все остальные задачи откладываются или прерываются.

## D

**DSL (Domain‑Specific Language)**  
Язык предметной области, используемый для написания политик. Позволяет описывать условия и действия декларативно.

**Data Flow (Поток данных)**  
Направление движения данных между компонентами системы (события, оценки, решения).

**Delayed Interrupt (Отложенное прерывание)**  
Прерывание, которое выполняется после заданной задержки, позволяя завершить текущие задачи.

## E

**Event (Событие)**  
Единица данных, представляющая произошедшее изменение или инцидент. Содержит поля `event_id`, `source`, `severity`, `urgency`, `impact`, `metadata`.

**Event Bus (Шина событий)**  
Инфраструктурный компонент (Kafka), обеспечивающий асинхронную передачу событий между компонентами.

**Elevated Mode (Повышенный режим)**  
Режим работы системы, в котором повышена внимательность к событиям, но не все задачи прерываются.

## F

**Feature Flag (Флаг функциональности)**  
Механизм включения/выключения функциональности в runtime, который может управляться через политики RAS.

## H

**Hard Interrupt (Жёсткое прерывание)**  
Немедленное прерывание текущей деятельности для обработки высокоприоритетного события.

**Hysteresis (Гистерезис)**  
Свойство системы, при котором порог перехода в другой режим зависит от текущего состояния (например, для перехода в elevated требуется score > 0.7, а для возврата — score < 0.5).

## I

**Impact (Влияние)**  
Одна из трёх осей scoring, оценивающая масштаб последствий события (0–1).

**Interrupt (Прерывание)**  
Временная остановка текущей деятельности для обработки более важного события.

**Interrupt Manager**  
Компонент, принимающий решение о прерывании на основе salience score и политик, управляющий чекпоинтами и восстановлением.

## K

**Kafka**  
Распределённая потоковая платформа, используемая как event bus в RAS.

## L

**Lag (Задержка)**  
Разница между временем публикации сообщения в Kafka и временем его обработки потребителем.

**Linear Combination (Линейная комбинация)**  
Метод вычисления salience score как взвешенной суммы severity, urgency и impact.

## M

**Metadata (Метаданные)**  
Дополнительные поля события, содержащие контекстную информацию (идентификаторы, метрики, теги).

**Mode (Режим)**  
Текущее состояние системы, определяющее её поведение: low, normal, elevated, critical.

**Mode Manager**  
Компонент, управляющий переходами между режимами на основе salience score и гистерезиса.

**ML Model (Машинно‑обучаемая модель)**  
Алгоритм, используемый Salience Engine для предсказания salience score на основе исторических данных.

## N

**Normal Mode (Нормальный режим)**  
Режим работы системы по умолчанию, в котором обрабатываются все события в порядке поступления.

## O

**Observability (Наблюдаемость)**  
Свойство системы, позволяющее понимать её внутреннее состояние на основе внешних выходных данных (логи, метрики, трассировки).

**OpenTelemetry**  
Стандарт для инструментации, генерации, сбора и экспорта телеметрии (трассировки, метрики, логи).

## P

**Partition (Партиция)**  
Логическое разделение топика Kafka для параллельной обработки.

**Policy (Политика)**  
Правило, состоящее из условия (condition) и действия (action), определяющее реакцию системы на событие.

**Policy Engine**  
Компонент, хранящий и исполняющий политики, предоставляющий API и веб‑интерфейс для управления.

**Prometheus**  
Система мониторинга и алертинга, используемая для сбора метрик RAS.

## R

**RAS (Reticular Activating System)**  
Ретикулярная активирующая система — биологическая концепция, вдохновившая архитектуру оркестратора: механизм selective attention и прерываний.

**Redis**  
Хранилище «ключ‑значение» in‑memory, используемое Workspace Service для хранения чекпоинтов, очередей задач и кэша.

**Retriever Agent**  
Компонент, который выполняет задачи, созданные Task Orchestrator, может интегрироваться с LLM и внешними системами.

**Rollback (Откат)**  
Процедура возврата системы к предыдущему состоянию после неудачного изменения.

## S

**Salience (Салентность)**  
Мера важности события, вычисляемая Salience Engine на основе severity, urgency, impact и дополнительных факторов.

**Salience Engine**  
Компонент, вычисляющий salience score для каждого события с использованием алгоритмов или ML‑моделей.

**Salience Score**  
Численное значение в диапазоне [0, 1], отражающее важность события.

**Severity (Серьёзность)**  
Одна из трёх осей scoring, оценивающая критичность события (0–1).

**Soft Interrupt (Мягкое прерывание)**  
Прерывание, которое позволяет завершить текущие задачи перед обработкой нового события.

**State Machine (Конечный автомат)**  
Модель, используемая Mode Manager для управления переходами между режимами.

## T

**Task (Задача)**  
Единица работы, создаваемая Task Orchestrator в результате прерывания. Имеет статус (pending, running, completed, failed).

**Task Orchestrator**  
Компонент, создающий задачи на основе решений Interrupt Manager и управляющий их жизненным циклом.

**Throughput (Пропускная способность)**  
Количество событий, обрабатываемых системой за единицу времени.

**Topic (Топик)**  
Именованный канал в Kafka, в который публикуются сообщения определённого типа (например, `ras.events`).

## U

**Urgency (Срочность)**  
Одна из трёх осей scoring, оценивающая временную критичность события (0–1).

## W

**Webhook**  
HTTP‑callback, используемый для интеграции с внешними системами (например, отправка уведомлений в Slack).

**Workspace Service**  
Компонент, предоставляющий абстракцию над Redis для хранения состояния системы, чекпоинтов и очередей задач.

## Z

**Zero‑Downtime Deployment (Развёртывание без простоя)**  
Стратегия обновления системы, при которой обслуживание не прерывается (blue‑green, canary).
