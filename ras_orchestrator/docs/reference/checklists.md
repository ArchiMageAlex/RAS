# Чеклисты

В этом документе собраны проверочные списки для различных сценариев: развёртывание, мониторинг, troubleshooting, безопасность.

## Чеклист развёртывания в production

### Предварительные требования

- [ ] Выбран регион и cloud provider (AWS, GCP, Azure, on‑prem).
- [ ] Созданы сети, security groups, IAM роли.
- [ ] Установлен Kubernetes кластер (версия 1.24+).
- [ ] Установлен Helm (если используется).
- [ ] Настроен Container Registry (ECR, GCR, Docker Hub) с доступом из кластера.

### Инфраструктура

- [ ] Kafka кластер развёрнут (минимум 3 брокера, replication factor 3).
- [ ] Redis кластер развёрнут (минимум 3 узла, persistence включена).
- [ ] PostgreSQL развёрнут (минимум primary + replica, backup настроен).
- [ ] Настроены persistent volumes для Kafka, Redis, PostgreSQL (если требуется).
- [ ] Настроены мониторинг и алертинг для инфраструктуры.

### Развёртывание RAS компонентов

- [ ] Собраны Docker‑образы для всех компонентов и загружены в registry.
- [ ] Созданы Kubernetes манифесты или Helm chart.
- [ ] Настроены ConfigMaps и Secrets (API ключи, пароли БД, TLS сертификаты).
- [ ] Применены манифесты в namespace `ras-production`.
- [ ] Проверены readiness и liveness пробы.
- [ ] Настроены Ingress/Route для API Gateway и Policy Engine UI.
- [ ] Включен TLS (HTTPS) для внешних endpoint.

### Настройка Observability

- [ ] Развёрнут Prometheus, Grafana, Loki, Jaeger (через Helm или операторы).
- [ ] Настроены service monitors для сбора метрик с RAS компонентов.
- [ ] Импортированы дашборды Grafana из `observability/grafana/dashboards/`.
- [ ] Настроены алерты в Alertmanager (правила из `observability/alert_rules.yml`).
- [ ] Проверена отправка логов в Loki (через Promtail или fluent‑bit).
- [ ] Проверена трассировка OpenTelemetry (трассировки появляются в Jaeger).

### Интеграции

- [ ] Настроены webhooks для уведомлений (Slack, PagerDuty, email).
- [ ] Настроены источники событий (Prometheus Alertmanager, внешние системы).
- [ ] Протестирована отправка тестового события и обработка.

### Тестирование

- [ ] Smoke‑тест: отправка события и проверка обработки.
- [ ] Нагрузочное тестирование: проверка пропускной способности и стабильности.
- [ ] Тестирование отказоустойчивости: убийство подов, отключение Kafka/Redis.
- [ ] Проверка backup и восстановления.

### Документация

- [ ] Обновлены конфигурации в Git.
- [ ] Задокументированы endpoint'ы, API ключи, процедуры.
- [ ] Созданы runbooks для типичных операций.

## Чеклист ежедневного мониторинга

### Системное здоровье

- [ ] Все поды в namespace `ras-production` в состоянии Running.
- [ ] CPU и память всех компонентов в пределах нормы (<80%).
- [ ] Kafka brokers и consumer groups здоровы.
- [ ] Redis доступен, использование памяти в норме.
- [ ] PostgreSQL доступен, нет long‑running queries.

### Бизнес‑метрики

- [ ] Количество событий (`ras_events_total`) соответствует ожидаемому.
- [ ] Salience score распределение нормальное (нет аномалий).
- [ ] Режим системы (`ras_current_mode`) — normal или elevated (не critical).
- [ ] Количество прерываний (`ras_interrupt_decisions_total`) в пределах нормы.
- [ ] Задачи завершаются успешно (failure rate <5%).

### Логи

- [ ] Нет ошибок уровня ERROR или CRITICAL в логах компонентов.
- [ ] Нет повторяющихся предупреждений.
- [ ] Логи поступают в Loki, доступны для поиска.

### Алерты

- [ ] Нет активных critical алертов в Alertmanager.
- [ ] Warning алерты расследованы и заведены задачи при необходимости.
- [ ] Проверены нотификации (Slack, email) на актуальность.

### Резервное копирование

- [ ] Backup PostgreSQL выполнен успешно (проверить время и размер).
- [ ] Snapshot Redis создан.
- [ ] Репликация Kafka работает (lag минимален).

## Чеклист troubleshooting

### Общие шаги

- [ ] Определить scope проблемы: какой компонент затронут?
- [ ] Собрать данные: логи, метрики, трассировки за период инцидента.
- [ ] Проверить недавние изменения (деплой, конфигурация, нагрузка).

### События не поступают

- [ ] Проверить health endpoint API Gateway.
- [ ] Проверить логи API Gateway на наличие ошибок.
- [ ] Убедиться, что Kafka топик `ras.events` существует и доступен.
- [ ] Проверить, что consumer (Salience Engine) работает и не отстаёт.
- [ ] Проверить API‑ключ и network policies.

### Высокий Kafka lag

- [ ] Определить, какой consumer group отстаёт.
- [ ] Проверить метрики CPU/памяти consumer.
- [ ] Увеличить количество partitions или реплик consumer.
- [ ] Проверить network latency между Kafka и consumer.

### Redis ошибки

- [ ] Проверить доступность Redis (`redis-cli ping`).
- [ ] Проверить использование памяти (близко к maxmemory?).
- [ ] Проверить количество соединений (не превышен ли limit?).
- [ ] Перезапустить Redis при необходимости.

### Политики не срабатывают

- [ ] Проверить, что политика enabled.
- [ ] Протестировать условие через API `/policies/test`.
- [ ] Проверить логи Policy Engine на ошибки парсинга.
- [ ] Убедиться, что событие содержит нужные поля.

### Система застряла в critical режиме

- [ ] Проверить salience score текущих событий.
- [ ] Проверить конфигурацию гистерезиса в Mode Manager.
- [ ] Принудительно перевести режим через API `/mode/force`.
- [ ] Исследовать, нет ли зацикленных политик.

### Производительность degraded

- [ ] Определить bottleneck через метрики latency.
- [ ] Проверить использование ресурсов (CPU, память, диск I/O).
- [ ] Увеличить ресурсы или масштабировать компонент.
- [ ] Оптимизировать запросы к БД, добавить индексы.

### Восстановление после сбоя

- [ ] Определить root cause.
- [ ] Применить fix (конфигурация, код, ресурсы).
- [ ] Протестировать восстановление (откат, restart).
- [ ] Обновить runbook и документацию.

## Чеклист безопасности

### Аутентификация и авторизация

- [ ] API Gateway требует API‑ключ для всех endpoint'ов, кроме health.
- [ ] API‑ключи хранятся в secrets manager, регулярно ротируются.
- [ ] Policy Engine имеет RBAC (если используется многопользовательский режим).
- [ ] Внутренние коммуникации используют mTLS или находятся в trusted network.

### Шифрование

- [ ] Все внешние endpoint'ы используют TLS (HTTPS).
- [ ] Данные в покое зашифрованы (диски БД, Redis persistence).
- [ ] Данные в движении зашифрованы (TLS для Kafka, Redis, PostgreSQL).

### Сетевая безопасность

- [ ] Компоненты изолированы network policies (Kubernetes).
- [ ] Доступ к административным интерфейсам (Grafana, Jaeger) ограничен по IP.
- [ ] Используются private subnets, нет публичного доступа к БД и Redis.

### Управление секретами

- [ ] Никакие секреты не хранятся в коде или конфигурационных файлах в Git.
- [ ] Secrets manager интегрирован с Kubernetes (например, External Secrets).
- [ ] Права доступа к секретам ограничены по принципу least privilege.

### Аудит и логирование

- [ ] Логируются все запросы к API Gateway (включая источник, API‑ключ, результат).
- [ ] Логируются изменения политик (кто, когда, что изменил).
- [ ] Логи защищены от несанкционированного изменения.
- [ ] Регулярно проводится аудит логов на подозрительную активность.

### Обновления и уязвимости

- [ ] Регулярно обновляются базовые образы контейнеров.
- [ ] Сканирование образов на уязвимости (Trivy, Snyk) интегрировано в CI/CD.
- [ ] Критические уязвимости исправляются в течение SLA (например, 72 часа).

### Соответствие стандартам

- [ ] Система соответствует внутренним security policies.
- [ ] Проведён penetration testing (если требуется).
- [ ] Документированы security controls для аудита.

## Чеклист обновления версии

### Подготовка

- [ ] Прочитаны release notes новой версии.
- [ ] Определены breaking changes и необходимые миграции.
- [ ] Создан backup всех данных (PostgreSQL, Redis, Kafka).
- [ ] Протестировано обновление в staging среде.

### План обновления

- [ ] Выбрана стратегия обновления (rolling, blue‑green, canary).
- [ ] Определено окно обновления (время низкой нагрузки).
- [ ] Уведомлены заинтересованные стороны (команды, пользователи).

### Выполнение

- [ ] Обновлены образы в Container Registry.
- [ ] Применены обновлённые манифесты (по одному компоненту).
- [ ] Мониторинг метрик во время обновления.
- [ ] Проведён smoke‑тест после обновления каждого компонента.

### Проверка

- [ ] Все поды запущены, readiness пробы прошли.
- [ ] Функциональность проверена (отправка события, политики, задачи).
- [ ] Производительность в норме (latency, throughput).
- [ ] Нет новых алертов.

### Откат (если требуется)

- [ ] Определены условия для отката (например, ошибки в течение 15 минут).
- [ ] Подготовлены предыдущие версии образов.
- [ ] Выполнен откат по компонентам.

### Документация

- [ ] Обновлены версии в документации.
- [ ] Задокументированы проведённые миграции.
- [ ] Обновлены runbooks.

## Чеклист инцидента

### Обнаружение

- [ ] Определён severity инцидента (critical, major, minor).
- [ ] Собрана initial информация: что сломалось, когда, кто обнаружил.
- [ ] Активирована команда ответственных.

### Диагностика

- [ ] Собраны логи, метрики, трассировки.
- [ ] Определён root cause.
- [ ] Оценено impact (сколько пользователей/сервисов затронуто).

### Восстановление

- [ ] Применены временные меры (restart, scale, disable feature).
- [ ] Применены постоянные меры (fix, конфигурация).
- [ ] Проверено восстановление функциональности.

### Коммуникация

- [ ] Уведомлены заинтересованные стороны о статусе.
- [ ] Предоставлены регулярные updates.
- [ ] После разрешения инцидента отправлен final report.

### Постмортем

- [ ] Проведена встреча для анализа инцидента.
- [ ] Задокументированы timeline, root cause, действия.
- [ ] Определены preventive меры.
- [ ] Созданы/обновлены runbooks.
- [ ] Заведены задачи для реализации мер.

## Чеклист вывода системы из эксплуатации

### Подготовка

- [ ] Уведомлены все пользователи и интеграции.
- [ ] Определена дата отключения.
- [ ] Создан план миграции данных (если требуется).

### Остановка

- [ ] Остановлен приём новых событий (отключены источники).
- [ ] Дождаться обработки всех pending событий и задач.
- [ ] Остановлены компоненты (scale to zero).
- [ ] Удалены Kubernetes ресурсы (deployments, services, configmaps).

### Архивация данных

- [ ] Экспортированы данные из PostgreSQL (дамп).
- [ ] Сохранены snapshots Redis.
- [ ] Сохранены логи и метрики за последний период.
- [ ] Данные перемещены в cold storage.

### Очистка

- [ ] Удалены persistent volumes.
- [ ] Удалены Container Registry образы (или помечены как deprecated).
- [ ] Удалены секреты из secrets manager.
- [ ] Обновлена документация (отметить систему как deprecated).

### Подтверждение

- [ ] Проверено, что нет зависимых систем.
- [ ] Уведомлены заинтересованные стороны о завершении.
- [ ] Закрыты связанные задачи и documentation.
